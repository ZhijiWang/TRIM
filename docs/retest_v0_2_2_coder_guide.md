# v0.2.2 Retest Coder Guide

## What You Are Coding

For each case, identify how selected textual evidence becomes one project-defined
analytic function. Code the analytic function assigned by the project, not
simply what an actor does in the story.

Actor uptake counts as the function only when action-guiding standing is the
central analytic role.

## Required Steps

All formal coding must be based on the supplied packet. The formal segments
provide source text or documented public-domain translation text. External links
are provenance references and are not required for completion.

1. Read the case metadata and source segments.
2. Check the permitted case scope.
3. Select one to three `primary_evidence_segment_ids`.
4. Add any `context_segment_ids` that support sequence or setting.
5. Fill `evidence_anchor`, `anchor_node`, and optional `evidence_highlight`.
6. Select one closed-list `function_label`.
7. Complete the six signature fields.
8. Calibrate `uncertainty_flag`.
9. Write `rationale_note`.
10. Add `alternative_signature` if a complete competing pathway remains viable.
11. Record any question in the question log.

## Evidence

Primary evidence is where the conversion turns. Context helps explain why those
segments matter. Do not mark every supplied segment primary unless each segment
is indispensable and your rationale explains why.

## Function Labels

Use only:

- `immediate_stabilization`
- `extended_deliberation`
- `ambition_trigger_authorization`
- `false_security`
- `retrospective_trap`
- `self_justification`
- `self_defence_self_accusation`
- `epistemic_suspension`
- `no_fit`

Use `no_fit` only when no substantive label can be supported.

## Uncertainty

- Low: one complete pathway is clearly preferable.
- Medium: one pathway is preferable but a complete alternative is defensible.
- High: the evidence or rules cannot stabilize the choice.

A complete plausible alternative normally means medium or high uncertainty.

## Questions

Log every question, including questions you resolve yourself. Mark whether it
was blocking or nonblocking and describe your provisional resolution.

Questions should be logged when they arise during coding, not reconstructed in a
batch after all cases are complete. Self-resolved questions must still be logged.
Timestamps should reflect when the issue arose. If exact timing is unavailable,
mark the timestamp as approximate rather than fabricating precision. If multiple
questions are batch-entered with identical timestamps, disclose that as a
protocol deviation.

Do not seek case-specific coaching before lock.

## Shared Context

Use only the context permitted in the manifest. If the case says
`cross_case_context_permitted=no`, do not use related cases to select the
signature. A `multi_passage_single_case` may contain separated passages from the
same formal case, but it does not permit cross-case context.
